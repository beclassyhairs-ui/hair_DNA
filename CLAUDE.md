# CLAUDE.md — A-Beauty (어뷰티) 작업 규칙

이 파일은 Claude Code가 매 세션 자동으로 따라야 하는 프로젝트 규칙이다.

## 0. 세션 시작 시 필수

1. `docs/PROJECT_STATE.md`와 `docs/ROADMAP.md`를 둘 다 읽는다. 작업 순서·정책은 ROADMAP.md를 따르고, 진척 상태는 PROJECT_STATE.md를 따른다.
2. `git status --short`와 `git log --oneline -5`로 실제 워크스페이스 상태를 확인한다.
3. PROJECT_STATE.md와 실제 상태가 다르면 사용자에게 보고하고 PROJECT_STATE.md를 먼저 갱신한다.

## 1. 프로젝트 정의

A-Beauty는 AI 헤어 진단 + 발견템 커머스 서비스다.
핵심은 UI가 아니라 데이터 흐름이다: 진단 → 모발 타입 → 상품 매칭 → 클릭 데이터 → 소싱/제휴/ODM.
수익 핵심은 커머스이며, 미용실 예약 연동은 하지 않는다.

## 2. 작업 흐름 (필수 준수)

모든 작업은 다음 순서로 진행한다:

1. 사용자 의도 확인 → 작업 계획 요약 제시
2. 구현
3. **백엔드/위험 작업이면 Codex 자동 검수** (아래 3번 규칙)
4. 결과 + 검수 요약 보고
5. 사용자 승인 후에만 커밋
6. 세션 종료 전 PROJECT_STATE.md 갱신

## 3. Codex 자동 검수 규칙

다음 경로의 파일을 **수정한 경우**, 커밋 제안 전에 반드시 Codex CLI 검수를 실행한다:

- `app/api/**` (모든 API 라우트)
- `supabase/**` (모든 SQL/스키마)
- `lib/supabaseAdmin.ts`, `lib/supabaseClient.ts`, `lib/products.ts`, `lib/sourcing.ts`
- `middleware.ts` (생성 시)
- 인증/권한/RLS/환경변수 관련 모든 변경

검수 실행 방법:

```bash
codex exec "다음 변경사항을 검수해줘. 범위: [변경 목적 한 줄]. 확인할 것: 1) 의도한 변경만 있는지 2) 보안 문제(인증 우회, 내부 필드 노출, RLS) 3) 기존 동작 파괴 여부. 결론을 '통과/수정 필요'로 답해줘. $(git diff)"
```

- 검수 결과는 원문 그대로가 아니라 **결론 + 문제점 요약**으로 사용자에게 보고한다.
- '수정 필요'가 나오면 수정 후 재검수한다. 커밋을 제안하지 않는다.
- 프론트엔드 전용 변경(UI 문구, 스타일, 컴포넌트 레이아웃)은 검수 생략 가능.

## 4. 절대 금지 (사용자 명시 승인 없이는 불가)

- Supabase SQL Editor 실행 (읽기 전용 사전점검 쿼리 포함 — 항상 사용자가 직접 실행)
- products 테이블 RLS anon SELECT 정책 추가
- /items를 DB에 직접 연결
- keep 클릭 즉시 products 저장 또는 /items 노출
- image_url 없는 상품 노출, uncertain 링크 자동 등록
- 상품 status를 코드에서 자동으로 'approved' 처리
- AI로 실제 상품 사진처럼 이미지 생성
- 커밋 / push / 배포 (매번 사용자 승인 필수)
- 환경변수 값 변경 또는 출력
- service_role 키를 서버 API 라우트 밖에서 사용

## 5. 커밋 규칙

- 작업 성격이 다르면 커밋을 분리한다 (기능/스키마/문서 섞지 않음)
- 커밋 메시지: `feat:` `fix:` `chore:` `docs:` 접두어 사용
- 커밋 전 반드시 `git status --short` 결과와 커밋 메시지를 사용자에게 보여주고 승인받는다

## 6. PROJECT_STATE.md 갱신 규칙

세션에서 의미 있는 작업(코드 변경, 커밋, 결정)이 있었으면 종료 전 `docs/PROJECT_STATE.md`를 갱신한다:

- "현재 상태 한 줄" 갱신
- 미커밋 변경 파일 목록
- 다음 작업 순서 (완료 항목 제거, 신규 항목 추가)
- 새로 발견한 위험/버그는 백로그에 추가

## 7. 소싱 인박스

- `sourcing/inbox/*.tsv` 에 새 파일이 있으면 리서치 에이전트가 넣은 상품 후보다.
- 사용자가 요청하면 파싱해서 요약을 보여준다. **자동으로 DB에 넣지 않는다.**
- 처리 완료된 파일은 `sourcing/processed/`로 이동한다.

## 8. 아키텍처 원칙

- 공개 조회는 반드시 서버 API 라우트 경유 (status='approved' + image_status='approved' 필터)
- 관리자 API와 공개 API는 별도 라우트로 분리
- 관리자 GET 응답은 필요한 필드만 명시 (select("*") 금지)
- 유저용 카카오 로그인과 관리자 인증은 완전 별개 — 서로 재사용 금지
- 신규 컬럼 기본값: status='draft', image_status='needs_review'

## 9. 톤 (사용자 대면 문구 작성 시)

- 타깃: 30~50대 여성, 하이엔드 뷰티 감성
- 전문적이지만 친절, 고급스럽고 따뜻하게
- 공포 조장 금지, 미용실 업셀링 느낌 금지

## 11. 큰 결정 교차검증

다음에 해당하는 결정을 사용자에게 제안하기 전에는, 먼저 Codex CLI에 "이 계획의 허점과 반대 논리를 찾아달라"고 요청해 반론을 받고, 그 반론을 반영하거나 반박한 최종안을 사용자에게 보고한다:

- 되돌리기 어려운 구조 변경 (DB 스키마, 인증 방식, 배포 구조)
- 돈/노출과 직결된 결정 (상품 노출 로직, 결제, 외부 연동)
- 두 가지 이상의 방안이 팽팽한 갈림길

보고 시 "Codex 반론 요지"를 한 줄 포함한다. 일상 작업에는 적용하지 않는다.

## 13. 토큰 절약 운영

- 작업 1건이 끝나면(커밋 + PROJECT_STATE.md 갱신 완료 후) 보고 마지막 줄에 "✅ 여기서 /clear 하셔도 안전합니다"를 반드시 표시한다.
- 세션 시작 시 PROJECT_STATE.md의 "다음 작업"만 읽고 바로 시작한다 — 과거 이력 재탐색 금지.
- 파일은 필요한 부분만 읽는다. 같은 파일 반복 읽기 금지.
- 단순 작업(라벨 수정, 문구 변경, 문서 갱신)은 사용자에게 "이건 소넷으로도 충분합니다"라고 알려준다.

## 14. 연속 진행 모드

사용자가 "이어서 진행해"라고 하면 ROADMAP 순서대로 다음 작업을 진행하되, 한 작업이 끝나도 멈추지 않는다. 커밋과 PROJECT_STATE.md 갱신까지 마친 뒤 바로 다음 작업으로 넘어간다.

사용자에게 묻는 것은 딱 세 경우만: ① push/배포 승인 ② 금지 규칙(4번)에 걸리는 결정 ③ 방안이 팽팽해 사업 판단이 필요할 때. 그 외엔 추천안대로 진행하고 결과만 쌓아서 보고한다.

한도가 가까워지면 진행 중 작업을 커밋 가능한 상태로 마무리하고 상태를 저장한 뒤 멈춘다.

# Compact instructions

압축(compact) 시 다음만 보존한다: 현재 진행 중인 작업의 목표와 남은 단계, 직전에 수정한 파일 경로 목록, 미해결 에러, PROJECT_STATE.md에 아직 반영 안 된 결정사항. 오래된 탐색 과정, 이미 커밋된 작업의 상세 로그, 파일 전체 내용은 버려도 된다.
