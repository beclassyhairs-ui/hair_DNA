# A-Beauty ROADMAP — 작업 큐 (PM방 ↔ Claude Code 연락 노트)

> 이 파일은 PM방(Claude 채팅)에서 확정된 작업 순서·검토 결론을 Claude Code에 전달하는 큐다.
> 세션 시작 시 `docs/PROJECT_STATE.md`와 함께 읽을 것. **진척·구현 상태가 충돌하면 PROJECT_STATE.md가 우선**이고, 순서·정책이 충돌하면 이 파일이 우선.
> 갱신 주체: PM방(사업주 승인 후). Claude Code는 이 파일을 직접 수정하지 않고, 완료 항목은 PROJECT_STATE.md에 기록한다.
> 마지막 갱신: 2026-07-19

## 0. 승인 관문 (2026-07-19 조정)

- 수정·커밋·일반 명령: 자동 승인 (settings.json). 파괴 명령(rm -rf, force push, reset --hard, clean -f)은 하드 차단.
- 사람 승인 3관문 유지: ① **push/배포 직전 확인** (push=Vercel 자동 배포이므로, CLAUDE.md 4번) ② **Supabase SQL은 사업주가 SQL Editor에서 직접 실행** (🟢조회/🔴변경 구분해 안내) ③ **상품 실노출(approved) 승인**.
- 백엔드/DB/인증/RLS/환경변수 변경은 커밋 전 Codex 검수 필수. '수정 필요' 판정이면 커밋 제안 금지.

## 1. 작업 순서 (위에서부터 순서대로)

1. **[진행 중] 최소 인증 게이트 마무리 + 배포 검증** — middleware.ts + ADMIN_SECRET + 서명 쿠키로 /admin·/api/admin 보호. 배포 후: 메인·/style 정상 + /admin 인증 요구 확인. (로그인 동작 확인됨 — 배포 검증까지 완료 시 PROJECT_STATE에 완료 기록)
2. **가벼운 픽스 묶음** (UI 커밋, Codex 검수 불필요·화면 확인으로 충분):
   - /my-diary 기장 라벨 한 칸 밀림 수정 (surveyData 기준: bob=단발, shoulder=어깨선)
   - 로딩 15초 강제 광고 대기 제거 — 합성 완료 즉시 결과 이동 (애드센스 정책 리스크)
   - 결과 페이지 확대 금지 제거 (maximum-scale=1.0, user-scalable=no 삭제)
   - 가독성: 본문·설명 15~16px 이상, 버튼 15~16px, 보조 문구 13~14px, 장식 라벨 최소 11~12px + 저대비 텍스트 대비 상향 (40~50대 타깃)
   - /home 가짜 개인화 고정 문구("오늘은 습도가 높아...") 제거 → 사실 기반 문구로
   - 브라우저 기본 alert/confirm → 토스트로 교체
   - 결과지 CTA 우선순위 정리: ① 저장·프로필 누적 ② 홈/발견템 연결 ③ 공유 ④ 재진단. 하단 고정 "알림 신청" CTA는 전략과 안 맞음 — 교체.
   - **개인정보 문구 정합(우선 높음)**: 랜딩 "개인정보 미저장" 표기 vs 셀카가 submit-diagnosis로 전송·저장되는 실동작 충돌. 문구 수정 또는 저장 로직 정비. "사진은 합성 후 즉시 파기" 원칙은 사업주와 확정 후 반영.
3. **커머스 파이프라인 완성** (백엔드 — 전부 Codex 검수 대상):
   - /admin/sourcing keep → "draft로 저장" 버튼 연결. 즉시 저장 금지·명시적 버튼. status='draft' + image_status='needs_review' 서버 강제.
   - /admin/products에 fit_hair_types 태그 편집 UI + "이 상품이 어떤 모발타입에게 노출되는지" 미리보기.
   - /items 공개 조회 API 신설: status='approved' AND image_status='approved' 필터, 반환 필드 allowlist(내부 필드 제외), 관리자 API와 완전 분리(/api/products).
   - /items DB 연동 + 유저 hairTags 매칭 + /items/[id] 상세(상세 최상단에 "당신의 진단 결과 → 이 상품" 추천 이유).
4. **게이트: 상품 1개 수동 엔드투엔드 완주** — 소싱→검수→승인→노출→클릭/구매를 사람 손으로 끝까지 1회. **이 완주 전 자동화 3단계 착수 금지.**
5. **faceswap 복원 + 84 레퍼런스 연동** — 신규 작성 금지, git 히스토리(2026-06-25 시점 app/api/hair-transform/route.ts)의 faceswap 호출·pickReferenceUrl 복원 + 84폴더 매핑. 착수 전 필수 조건 5개: ① 실손님 셀카 유형별(안경/옆머리/역광) 품질 테스트 후 사업주 승인 ② 84개 밖 조합 폴백 규칙 ③ 레퍼런스 인물 유사성 점검 ④ 얼굴 데이터 외부 처리 개인정보 문구(업체 약관·보관기간) ⑤ 일일 호출 제한(lib/dailyLimit.ts 실제 적용 확인).
6. **카카오 OAuth 서버사이드 인증** — 인가코드→서버 토큰 교환→카카오 ID 확보. 현재 클라이언트 SDK+localStorage 플래그 방식은 서버가 유저 식별 불가(장식 상태). 유저 인증과 관리자 인증은 완전 별개 시스템 유지.
7. **profiles/diagnoses 테이블 서버 저장** — 모발 프로필 자산화. 이후 데이터 통합(진단 Sheets+Blob / 프로필 localStorage / 이벤트 Supabase 3곳 분산 → Supabase 중심, Sheets는 백업).

## 2. 자동화 단계 정의 (사업주 합의)

- 1단계(완료): AI 팀 체계 — Claude Code+Codex 연동, CLAUDE.md 규칙.
- 2단계(진행 중): 소싱 파이프라인 수동 완성 — 위 작업 순서 1~4.
- 3단계: sourcing_candidates 테이블 + 인세인서치 자동 투입(하루 1회 10~20개, 그 이상 금지). 자동화 대상은 "운반"뿐 — keep/drop 판단·승인·노출은 전부 사람.
- 4단계: 오픈클로 정기 점검 — 등록 상품 URL 생존·이미지 깨짐 순찰. 문제 발견 시 자동으로 내리지 않고 "검토 필요" 플래그 + 사업주 보고.
- 5단계: 데이터 루프 — 주간 자동 리포트(클릭 톱/클릭 0/타입별 전환율) → 정리 후보 제시 → 소싱 키워드 역반영.

## 3. 나중 실험 (지금 반영 금지)

셀카 옵션화 A/B(사진 없이 리포트 먼저 vs 현행 — /style 핵심 체험은 얼굴 합성이 확정이므로 구조 변경 아닌 실험으로만) · 전용 공유 결과 카드(타입 별명+한 줄 진단+로고+주소, 얼굴 포함 여부 유저 선택) · 카카오 채널 친구 추가+마케팅 수신 동의(로그인·개인정보 작업에 포함, 절차 분리 설계) · 실제 테스트 고객 후기 영역(가짜 후기 금지·동의 필수) · 톤 통일(배경색 통일이 아니라 폰트·카드·버튼·여백·대비 규칙 통일).

## 4. 문구·표기 규칙

- 서비스 노출 문구에 미검증 숫자 금지. 전문가 크레딧은 "현직 헤어디자이너가 직접 설계" 형태 — 연차 숫자는 사업주 확정 전 사용 금지.
- 절대 금지 목록은 CLAUDE.md를 따른다 (승인 없는 SQL/push/배포, RLS anon SELECT, keep 즉시 저장, status 자동 approved, AI 상품사진 생성, 인터넷 이미지 무단 사용, 유저 카카오 로그인의 관리자 인증 재사용, Gemini 무검수 등록).
